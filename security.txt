StayAnalytics — Password storage and authentication
====================================================

Goal
----
Never store a user's plaintext password. Even if the database dump
leaks, recovering the original password should require infeasible
brute force.

Approach: BCrypt (via Spring Security)
--------------------------------------
1. The application uses Spring Security's `BCryptPasswordEncoder`
   configured with a work factor (cost) of 12. The bean is declared in
   `SecurityConfig#passwordEncoder()`:

       @Bean
       public PasswordEncoder passwordEncoder() {
           return new BCryptPasswordEncoder(12);
       }

2. On signup (POST /signup), `UserService#register` calls
   `passwordEncoder.encode(rawPassword)` before any data leaves the
   service layer. The resulting hash is the only thing inserted into
   `users.password_hash` (column type VARCHAR(72)).

3. On login (POST /login), Spring Security's `DaoAuthenticationProvider`
   pulls the row by username through `JdbcUserDetailsService`
   (which executes Q10) and delegates verification to the same
   `BCryptPasswordEncoder.matches(rawPassword, storedHash)`. Plaintext
   passwords are never logged.

4. The hashes look like `$2a$12$<22-char salt><31-char hash>` (or
   `$2b$12$...` for hashes generated by the Python `bcrypt` library
   used to seed demo accounts in data.sql; both formats are accepted).

Why BCrypt at strength 12?
--------------------------
- BCrypt embeds a per-user random salt inside the hash, so two users
  with the same password get different stored values, defeating
  precomputed rainbow tables.
- Work factor 12 means 2^12 internal key-schedule iterations. On
  modern hardware this is ~250 ms per hash, slow enough to make brute
  forcing impractical while keeping login latency acceptable.
- BCrypt is adaptive: when hardware gets faster, the cost can be
  raised (e.g. to 13) without breaking existing hashes.

Other defensive measures in use
-------------------------------
- All SQL statements are executed via JDBC `PreparedStatement` with
  parameter binding, so user input cannot be concatenated into a
  query string. This eliminates the SQL-injection vector.
- Spring Security CSRF protection is enabled by default for all
  state-changing endpoints (`/signup`, `/login`, `/logout`,
  `/movies/*/reviews`, `/watchlist/*`).
- Session cookies are HttpOnly by default; passwords are never
  included in error messages or log output.
- The signup form rejects passwords shorter than 6 characters
  (server-side check in `UserService#register`).

Demo accounts (seeded in data.sql)
----------------------------------
   admin / Demo1!   (role ADMIN)
   alice / Demo2!   (role USER)
   bob   / Demo3!   (role USER)

The plaintext passwords are recorded here only to support grading;
they are never stored on the server side, only their BCrypt hashes.
